Two-Layer Processing Architecture
This is critical for performance given the ~400K row dataset. Do NOT query raw data at render time.
Layer 1 — Raw Interaction Storage (write once)

Store the uploaded Eloqua rows in a journey_raw_interactions table
Each row = one contact × one asset × one interaction
This table is written once during upload and never queried directly by the UI

Layer 2 — Pre-Computed Summary Tables (built during upload processing)
After raw data is stored, run a background processing job that computes and stores:
journey_summary_stats
├── unique_contacts (COUNT DISTINCT Contact Id)
├── total_interactions (COUNT *)
├── journey_patterns (COUNT DISTINCT ordered sequences)
├── stage_transitions (COUNT DISTINCT stage-pair combinations)
└── last_updated (timestamp)

journey_stage_flows
├── from_asset_id
├── from_asset_name
├── from_stage (TOFU/MOFU/BOFU)
├── to_asset_id
├── to_asset_name
├── to_stage (TOFU/MOFU/BOFU)
├── contact_count (how many contacts made this transition)
└── avg_days_between (average time between the two interactions)

journey_asset_summary
├── asset_id
├── asset_name
├── asset_type
├── funnel_stage (TOFU/MOFU/BOFU/Unclassified)
├── unique_contacts
├── total_interactions
├── entry_count (contacts who started their journey here)
├── exit_count (contacts whose journey ended here — i.e., drop-offs)
└── pass_through_count (contacts who continued to another asset after this one)

journey_top_paths
├── path_sequence (JSON array of ordered asset names)
├── path_stages (JSON array of corresponding stages)
├── contact_count
└── conversion_flag (boolean: did this path reach BOFU?)
Funnel Stage Classification
Assets are classified into funnel stages based on their Asset Name prefix or pattern matching. The classification rules (based on the current data):

BOFU: Asset name starts with BOFU — purchase, verification, payment flows
MOFU: Asset name starts with MOFU or contains consideration-stage keywords
TOFU: Asset name starts with TOFU — awareness, newsletter, introductory content
Unclassified: Anything that doesn't match the above patterns

Store classification rules in a configurable lookup so admins can reclassify assets without re-uploading data.

Processing Pipeline (Upload Flow)
When an admin uploads the Eloqua Excel file:
Step 1: Validation

Check required columns exist
Validate date formats
Report row count and any parsing errors
Show preview of first 10 rows

Step 2: Raw Storage

Insert all rows into journey_raw_interactions
Deduplicate by Contact Id + Asset Id + Activity Date (same contact, same asset, same timestamp = one interaction)

Step 3: Pre-Compute Summaries
Show a progress indicator during this step — it will take time with 400K rows.

Classify assets into funnel stages using the prefix rules
Build contact journeys: For each unique contact, order their interactions by Activity Date to create a sequence
Compute transitions: For each consecutive pair in a contact's journey, record the from→to transition
Aggregate into summary tables:

journey_stage_flows: GROUP BY (from_asset, to_asset), COUNT DISTINCT contacts
journey_asset_summary: Per-asset aggregates
journey_top_paths: GROUP BY full path sequence (cap at first 6 steps to avoid explosion), COUNT contacts
journey_summary_stats: Global aggregates


Update timestamp and flag processing as complete

Step 4: Refresh UI

Reload the Content Journey Mapping page with new data
Display "Last updated" timestamp


Technical Notes

Database: Use the existing shared persistent database (not SQLite in-memory — must be visible in deployed browser, not just Replit editor)
D3 Sankey: Use d3-sankey from CDN or npm. The d3-sankey plugin provides sankey() layout generator that handles node positioning and link path computation
Performance: The Sankey diagram should render from pre-computed summary tables only. Never query raw 400K rows for UI rendering.
Export: Add a "Download as PDF" button that exports the current Sankey view using the Sage brand template (black background, green accents, white text) — same pattern as existing comparison report PDFs
Empty state: If no Eloqua data is uploaded yet, show an empty state with upload CTA and a brief explanation of what this section does


What This Section Does NOT Do

Does NOT create content — this is purely analytical
Does NOT connect to Eloqua in real-time — data comes via Excel upload only
Does NOT show individual contact PII — all views are aggregated
Does NOT predict future journeys — it maps historical paths only
Does NOT replace the existing Content Library or Campaign Planner — it complements them with journey context
