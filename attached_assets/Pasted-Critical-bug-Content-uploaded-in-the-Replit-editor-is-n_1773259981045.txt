Critical bug: Content uploaded in the Replit editor is not visible in the external deployed app. Users see different content libraries depending on how they access the app. Some content shows in Replit but not in the external browser. Green dots for uploaded content are missing externally. MOFU and BOFU content that exists in Replit shows "No content found" externally. Fix this immediately — the content library must be identical for all users regardless of how they access the app.

DIAGNOSE THE ROOT CAUSE
This is almost certainly a storage/database issue. Check these in order:
Check 1 — Where is content_stored data being saved?

Is the content_stored table using an in-memory database (like SQLite in memory) that resets on each deployment or connection?
Is it using a file-based SQLite database stored in a non-persistent directory?
Is it using Replit's built-in database, an external database (Supabase, PostgreSQL), or local file storage?
The fix: content_stored must use a persistent database that is accessible from all instances of the app — both the Replit editor preview and the external deployed URL. Use one of these:

Replit's built-in Key-Value database (if using Replit DB)
An external database like Supabase, PlanetScale, Neon, or any hosted PostgreSQL
Replit's persistent /data directory for file storage (but database records must still be in a shared DB)


Do NOT use: in-memory storage, session storage, localStorage, or file-based SQLite in a non-persistent directory

Check 2 — Where are uploaded files stored?

Are the actual PDF/DOCX files stored in a directory that persists across deployments?
Replit's file system outside of /data or designated persistent directories can be wiped on redeployment
The fix: All uploaded content files must be stored in a persistent location:

Option A: Replit's persistent /data/content_files/{asset_id}/ directory
Option B: An external file storage service (S3, Cloudflare R2, Supabase Storage)
Option C: Store the file content as a blob or base64 in the database itself (works for smaller files)


Verify the files are actually accessible from the external deployed URL by checking the file path

Check 3 — Is the engagement data (Excel) loading differently?

The MOFU showing 0 assets and BOFU showing 0 assets externally but content existing in Replit suggests the engagement data Excel may also not be loading properly in the deployed version
Check if the Excel data is being read from a file path that only exists in the Replit editor
The fix: The engagement data should be loaded from the same persistent database as content_stored. When admin uploads an Excel, parse it and store the data in the database — don't read from the Excel file at runtime

Check 4 — Are there separate database instances?

Sometimes Replit editor and the deployed app connect to different database instances
Verify both environments point to the exact same database connection string
Log the database connection details in both environments and compare
